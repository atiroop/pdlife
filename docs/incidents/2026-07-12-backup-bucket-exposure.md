## Incident: ไฟล์ backup ฐานข้อมูลหลุดไปอยู่ใน public bucket ชั่วคราว

วันที่เกิดเหตุ: 12 กรกฎาคม 2569 เวลา 18:02 น.
วันที่แก้ไขเสร็จสมบูรณ์: 12 กรกฎาคม 2569 เวลา 18:34 น.

รายละเอียด: ระหว่างทดสอบฟีเจอร์ automated backup (cmd/db_backup) ครั้งแรก โค้ดยังอ่าน
R2_BUCKET (public bucket ที่เชื่อมกับ cdn.pdlife.app) แทนที่จะอ่าน R2_BACKUP_BUCKET
เนื่องจาก env var ตัวใหม่ยังไม่ถูกตั้งค่า ณ ขณะนั้น ทำให้ไฟล์ pdlife_backup_20260712.sql.gz
(มีข้อมูลสุขภาพผู้ใช้ในฐานข้อมูล) เปิดให้เข้าถึงผ่าน URL สาธารณะได้ชั่วคราว

การแก้ไข:
1. ลบไฟล์ต้นฉบับออกจาก R2 bucket สาธารณะ (ภายในเวลาไม่ถึง 1 นาทีหลังตรวจพบ)
2. Purge Cloudflare cache ของ URL ที่เกี่ยวข้อง — ยืนยันแล้วว่า URL คืนค่า 404
   (Cf-Cache-Status: EXPIRED) หลัง purge
3. แก้โค้ดให้ BackupConfigFromEnv() อ่านเฉพาะ R2_BACKUP_BUCKET ไม่ fallback ไป bucket สาธารณะ
   หากตั้งค่าไม่ครบ ระบบจะหยุดทำงานพร้อมแจ้งเตือนแทนการอัปโหลดผิดที่แบบเงียบๆ
4. ตรวจสอบทั่วทั้งระบบ ยืนยันไม่มีไฟล์อ่อนไหวอื่นหลุดไปที่เดียวกัน (27 objects audit)

ผลกระทบที่ประเมิน: ไม่มีหลักฐานว่ามีบุคคลอื่นนอกจากเจ้าของโปรเจกต์เข้าถึงไฟล์นี้ระหว่างที่เปิดอยู่
ระยะเวลาเปิดสาธารณะ: ประมาณ 31 นาที (18:02–18:34 น.) ผ่าน Cloudflare CDN edge cache — ตัวไฟล์บน
R2 origin เองถูกลบภายในเวลาไม่ถึง 1 นาทีหลังตรวจพบ แต่ CDN เคยดึงไฟล์ไปแคชไว้ก่อนลบ (max-age
14400 วินาที) จึงยังคงให้ดาวน์โหลดต่อได้จน purge cache สำเร็จ

บทเรียน: การเพิ่ม env var ใหม่ (R2_BACKUP_BUCKET) ควรตั้งค่าให้ครบก่อนรันโค้ดที่พึ่งพา
env var นั้นเสมอ ไม่ควรทดสอบ feature ที่เกี่ยวกับข้อมูลอ่อนไหวก่อนยืนยัน config ครบถ้วน
