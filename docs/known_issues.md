# Known Issues

## PDF Export — go-pdf/fpdf Thai glyph rendering ผิด (อัปเดต 2026-07-21: ร้ายแรงกว่าที่เคยเข้าใจ)

**อัปเดตสำคัญ (2026-07-21):** ตอนแรกเข้าใจว่าบั๊กนี้กระทบแค่ ToUnicode CMap (text
layer สำหรับ copy/search/screen reader) โดยที่ "การแสดงผลถูกต้อง 100%" — ระหว่างพัฒนา
export ของ Lab Results พบว่า**ไม่จริง** สตริงไทย "ลบ" (negative) render ออกมาเป็น
"au" (ตัวอักษรละติน) ให้เห็นตรงๆ บนหน้า PDF ที่ rasterize แล้ว ไม่ใช่แค่ text layer
ที่มองไม่เห็น — สร้างซ้ำได้แน่นอนด้วย isolated repro (fpdf + font ตัวเดียวกัน,
string "ลบ" เปล่าๆ) แต่ที่แปลกคือ**ไม่ deterministic ข้ามบริบท** — "ลบ" ที่อยู่ในเอกสาร
ที่มีตัวอักษรไทยชุดอื่นประกอบด้วย (เช่น isolated test ที่มี 12 คำ) กลับ render ถูก
ส่วนเอกสารที่มีตัวอักษรน้อยกว่ากลับพัง ชี้ว่าเป็นบั๊กใน font subsetting/glyph-ID
assignment ของ fpdf ที่ไวต่อ "ชุดตัวอักษร unique ทั้งหมดในเอกสารนั้น" ไม่ใช่ตัวอักษรตัวใดตัวหนึ่งเดี่ยวๆ
— **จึงคาดเดาล่วงหน้าไม่ได้ว่าคำไหนจะพังในเอกสารไหน** ต้อง render จริงแล้วตรวจด้วยตาเท่านั้น

ผลกระทบจริง (แก้ไขคำอธิบายเดิม): การแสดงผล PDF **ส่วนใหญ่**ถูกต้อง (ยืนยันด้วย
pdftoppm/PyMuPDF rasterize หลายกรณี — หัวตาราง, วันที่, ป้ายกำกับส่วนใหญ่ของ
APD/CAPD/HD ถูกต้อง) แต่**มีบางคำที่ render ผิดแบบมองเห็นได้ ขึ้นกับบริบท** ซึ่งอันตราย
กว่าที่บันทึกไว้เดิมมาก โดยเฉพาะกับข้อมูลทางคลินิกที่ต้องอ่านถูกต้อง (เช่น
ผล serology HBsAg/Anti HCV/Anti HIV เป็น negative/positive)

การลดความเสี่ยงที่ทำไปแล้ว (2026-07-21): `internal/handler/lab_results_export.go`
เปลี่ยนป้าย negative/positive ในไฟล์ PDF จากไทย ("ลบ"/"บวก" — พิสูจน์แล้วว่าพัง)
เป็นอังกฤษ ("Negative"/"Positive" — ASCII ล้วน ไม่มีความเสี่ยง) เฉพาะใน PDF
เท่านั้น หน้าเว็บ /lab-results ยัง render ผ่าน browser font engine ตามปกติ
ไม่กระทบ **ยังไม่ได้แก้ช่องโหว่ที่เหลือ**: ฟิลด์ free-text ที่ผู้ใช้พิมพ์เอง
(APD remark, HD notes, Lab CXR/EKG/Notes) ยังผ่าน fpdf pipeline เดิม —
ข้อความเหล่านี้เป็น user input เปลี่ยนเป็นอังกฤษให้ไม่ได้ จึงยังมีความเสี่ยงเดิม
ที่ยังไม่ปิดสนิท (ความเสี่ยงนี้มีอยู่ก่อนหน้านี้แล้วตั้งแต่ APD export วันที่ 2026-07-08
ไม่ใช่ของใหม่ เพียงแต่เพิ่งค้นพบความรุนแรงที่แท้จริง)

สาเหตุ: บั๊กใน library go-pdf/fpdf (utf8fontfile.go) ทุกเวอร์ชัน v0.1.0–v0.9.0
เป็น unexported field เข้าถึงจากนอก package ไม่ได้ ต้อง fork ถึงจะแก้ได้

แนวทางแก้ระยะยาว (ยังไม่ทำ): เปลี่ยนวิธี generate PDF เป็น render HTML ผ่าน
headless Chrome หรือ wkhtmltopdf แทนการใช้ go-pdf/fpdf โดยตรง — ตอนนี้ยิ่งสำคัญขึ้น
เพราะกระทบการแสดงผลจริง ไม่ใช่แค่ accessibility

Priority: **Medium** (ปรับขึ้นจาก Low) — ไม่ใช่บั๊กที่เห็นได้ทุกครั้ง แต่เมื่อเกิดขึ้น
